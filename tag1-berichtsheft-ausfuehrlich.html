Archiv – ausführliche Fassung vor der Kürzung, nur zum Vergleich
Tag 1 · Team Berichtsheft

Euer Auftrag: das Berichtsheft-Modell

AzubiHub ist das Ausbildungsportal, an dem ihr mitbaut. Euer Team ist heute für alles zuständig, was mit dem digitalen Berichtsheft zu tun hat: Berufsschulwochen, Praxiseinsatz-Wochen und Seminar-/Workshop-Wochen. Ihr bekommt kein fertiges Gerüst. Ihr lest ein Klassendiagramm und baut daraus selbst Klassen, eine abstrakte Klasse und ein Interface – Schritt für Schritt, in der Reihenfolge dieses Dokuments.

Bevor ihr anfangt: zwei Pfeile, die ihr auseinanderhalten müsst

Im Diagramm unten kommen zwei Pfeilarten mit einem hohlen Dreieck vor. Sie sehen ähnlich aus, bedeuten aber etwas komplett anderes:

Vererbung (extends) – die Unterklasse erbt alle Attribute und Methoden der Oberklasse und kann sie ergänzen oder überschreiben.
Interface-Umsetzung (implements) – die Klasse verspricht, alle Methoden des Interfaces bereitzustellen. Sie erbt dabei nichts.
Assoziation – die Klasse hält eine Referenz auf eine andere Klasse als Attribut.

Teil 1 – Die Basis: Person, Azubi, Ausbilder, Bpu

Diesen Teil baut ihr gemeinsam mit Team Praxiseinsatz nach demselben Diagramm. Beide Teams brauchen dieselbe Grundlage, deshalb bauen beide sie einmal selbst durch.

1

Person

Abstrakte KlasseLeicht

Legt die Klasse Person an und markiert sie als abstract. Es gibt in AzubiHub keine „Person ohne genauere Rolle" – jede echte Person ist Azubi, Ausbilder oder BPU. Der Konstruktor setzt alle drei Attribute direkt, ohne Prüfung. getRollenbezeichnung() bekommt keinen Rumpf, nur ein Semikolon nach der Signatur – sie ist abstract.

- personalnummer: int
- vorname: String
- nachname: String
+ Person(personalnummer, vorname, nachname)
+ getVollstaendigerName(): String
{abstract} + getRollenbezeichnung(): String
Eine abstrakte Methode hat keinen Rumpf. Genau das zwingt jede Unterklasse, eine eigene Antwort zu liefern – der Compiler lässt euch sonst nicht kompilieren.
2

Azubi

Klasse extends PersonLeicht

Azubi erbt von Person. Der Konstruktor nimmt zusätzlich ausbildungsjahr und fachrichtung entgegen und ruft als erste Anweisung super(personalnummer, vorname, nachname) auf. Überschreibt getRollenbezeichnung() so, dass sie "Azubi" zurückgibt. Das Enum Fachrichtung ist bereits vorgegeben, ihr müsst es nicht selbst bauen.

- ausbildungsjahr: int
- fachrichtung: Fachrichtung
+ Azubi(personalnummer, vorname, nachname, ausbildungsjahr, fachrichtung)
+ getRollenbezeichnung(): String
3

Ausbilder

Klasse extends PersonSehr leicht

Ausbilder erbt von Person und hat keine eigenen zusätzlichen Attribute. Der Konstruktor reicht alle drei Parameter einfach an super(...) weiter. Überschreibt getRollenbezeichnung() mit "Ausbilder".

+ Ausbilder(personalnummer, vorname, nachname)
+ getRollenbezeichnung(): String
Nicht jede Unterklasse braucht neue Attribute. Manchmal ist der einzige Unterschied, was getRollenbezeichnung() zurückgibt.
4

Bpu

Klasse extends PersonSehr leicht

Genau wie Ausbilder, nur dass getRollenbezeichnung() "BPU" zurückgibt.

+ Bpu(personalnummer, vorname, nachname)
+ getRollenbezeichnung(): String

Teil 2 – Euer Modell: Berichtshefte

Ab hier baut ihr allein weiter, in eurem eigenen Package. Achtet auf die Reihenfolge: Enums und Interface zuerst, dann die abstrakte Klasse, dann die drei konkreten Unterklassen.

5

Berichtsart

EnumLeicht

Legt das Enum Berichtsart mit genau drei Konstanten an.

BERUFSSCHULE
PRAXISEINSATZ
SEMINAR_WORKSHOP
6

Berichtsstatus

EnumLeicht

Legt das Enum Berichtsstatus mit genau drei Konstanten an. Das ist bewusst eine vereinfachte Fassung – der echte Ablauf mit BPU-Zwischenschritt kommt erst an Tag 2 dazu.

ENTWURF
EINGEREICHT
FREIGEGEBEN
7

Protokollierbar

InterfaceLeicht

Legt das Interface Protokollierbar mit genau einer Methode an. Jede Klasse, die dieses Interface implementiert, verspricht eine Methode verlauf(), die eine Liste mit lesbaren Text-Zeilen liefert. Team Praxiseinsatz baut in seinem Bereich dasselbe Interface. Später soll die Oberfläche darüber für zwei Klassen, die sonst nichts miteinander zu tun haben, eine einheitliche „Verlauf"-Ansicht zeigen können.

+ verlauf(): List<String>
8

Berichtsheft

Abstrakte Klasse, implements ProtokollierbarMittel

Legt Berichtsheft als abstract an und lasst sie Protokollierbar implementieren. Der Konstruktor setzt alle vier Parameter direkt und zusätzlich status = Berichtsstatus.ENTWURF, obwohl status gar kein Parameter ist – jedes neue Berichtsheft startet als Entwurf. getArt() ist abstract, jede Unterklasse muss sie ausfüllen. brauchtBpuFreigabe() bekommt dagegen schon einen Rumpf: return false; – das ist der Normalfall. Für verlauf() reicht an Tag 1 eine Liste mit einem einzigen Eintrag, zum Beispiel List.of(getArt() + " – KW" + kalenderwoche + " – " + status). Eine echte Historie kommt an Tag 2.

- nummer: int
- azubi: Azubi
- kalenderjahr: int
- kalenderwoche: int
- status: Berichtsstatus
+ Berichtsheft(nummer, azubi, kalenderjahr, kalenderwoche)
{abstract} + getArt(): Berichtsart
+ brauchtBpuFreigabe(): boolean
+ verlauf(): List<String>
Wichtig: brauchtBpuFreigabe() ist nicht abstract – sie hat schon einen Rumpf. Trotzdem darf eine Unterklasse sie überschreiben, wenn sie ein anderes Verhalten braucht. Das ist der Unterschied zu getArt(): dort muss überschrieben werden, hier kann.
9

BerufsschulBerichtsheft

Klasse extends BerichtsheftLeicht

Ruft im Konstruktor super(nummer, azubi, kalenderjahr, kalenderwoche) auf und überschreibt getArt() so, dass sie Berichtsart.BERUFSSCHULE zurückgibt.

+ BerufsschulBerichtsheft(nummer, azubi, kalenderjahr, kalenderwoche)
+ getArt(): Berichtsart
10

PraxiseinsatzBerichtsheft

Klasse extends BerichtsheftLeicht–Mittel

Wie BerufsschulBerichtsheft, aber getArt() gibt Berichtsart.PRAXISEINSATZ zurück. Zusätzlich überschreibt diese Klasse als einzige auch brauchtBpuFreigabe() und gibt true zurück – nur Praxiseinsatz-Berichte brauchen die BPU-Station im Ablauf.

+ PraxiseinsatzBerichtsheft(nummer, azubi, kalenderjahr, kalenderwoche)
+ getArt(): Berichtsart
+ brauchtBpuFreigabe(): boolean
Das ist die einzige Stelle in eurem Modell, an der eine konkrete (nicht-abstrakte) geerbte Methode überschrieben wird, nicht nur eine abstrakte.
11

SeminarWorkshopBerichtsheft

Klasse extends BerichtsheftLeicht

Wie BerufsschulBerichtsheft, aber getArt() gibt Berichtsart.SEMINAR_WORKSHOP zurück.

+ SeminarWorkshopBerichtsheft(nummer, azubi, kalenderjahr, kalenderwoche)
+ getArt(): Berichtsart

Wann seid ihr fertig?

  • Alle elf Typen aus diesem Dokument existieren mit exakt diesen Namen.
  • Das Projekt kompiliert ohne Fehler.
  • getArt() ist in allen drei Unterklassen überschrieben, @Override steht dabei.
  • PraxiseinsatzBerichtsheft überschreibt zusätzlich brauchtBpuFreigabe().
  • Ihr könnt an eurem eigenen Diagramm zeigen, welcher Pfeil Vererbung und welcher Interface-Umsetzung ist.
  • Mindestens eine Person im Team kann erklären, warum Berichtsheft abstrakt ist, aber BerufsschulBerichtsheft nicht.
Tag 2: Ihr bekommt eine geprüfte, ergänzte Fassung dieses Modells zurück – mit echter Statushistorie und dem vollen fünfstufigen Freigabeablauf. Darauf arbeitet ihr weiter, unabhängig davon, wie weit ihr heute gekommen seid.

Tag 1 · Team Praxiseinsatz

Euer Auftrag: das Praxiseinsatz-Modell

AzubiHub ist das Ausbildungsportal, an dem ihr mitbaut. Euer Team ist heute für alles zuständig, was mit der Planung von Praxiseinsätzen zu tun hat: Abteilungen, Praxisangebote, Praxisplätze und die konkrete Zuweisung eines Azubis. Ihr bekommt kein fertiges Gerüst. Ihr lest ein Klassendiagramm und baut daraus selbst Klassen und ein Interface – Schritt für Schritt, in der Reihenfolge dieses Dokuments.

Bevor ihr anfangt: zwei Pfeile, die ihr auseinanderhalten müsst

Im Diagramm unten kommen zwei Pfeilarten mit einem hohlen Dreieck vor. Sie sehen ähnlich aus, bedeuten aber etwas komplett anderes:

Vererbung (extends) – die Unterklasse erbt alle Attribute und Methoden der Oberklasse und kann sie ergänzen oder überschreiben.
Interface-Umsetzung (implements) – die Klasse verspricht, alle Methoden des Interfaces bereitzustellen. Sie erbt dabei nichts.
Assoziation – die Klasse hält eine Referenz auf eine andere Klasse als Attribut.

Teil 1 – Die Basis: Person, Azubi, Ausbilder, Bpu

Diesen Teil baut ihr gemeinsam mit Team Berichtsheft nach demselben Diagramm. Beide Teams brauchen dieselbe Grundlage, deshalb bauen beide sie einmal selbst durch.

1

Person

Abstrakte KlasseLeicht

Legt die Klasse Person an und markiert sie als abstract. Es gibt in AzubiHub keine „Person ohne genauere Rolle" – jede echte Person ist Azubi, Ausbilder oder BPU. Der Konstruktor setzt alle drei Attribute direkt, ohne Prüfung. getRollenbezeichnung() bekommt keinen Rumpf, nur ein Semikolon nach der Signatur – sie ist abstract.

- personalnummer: int
- vorname: String
- nachname: String
+ Person(personalnummer, vorname, nachname)
+ getVollstaendigerName(): String
{abstract} + getRollenbezeichnung(): String
Eine abstrakte Methode hat keinen Rumpf. Genau das zwingt jede Unterklasse, eine eigene Antwort zu liefern – der Compiler lässt euch sonst nicht kompilieren.
2

Azubi

Klasse extends PersonLeicht

Azubi erbt von Person. Der Konstruktor nimmt zusätzlich ausbildungsjahr und fachrichtung entgegen und ruft als erste Anweisung super(personalnummer, vorname, nachname) auf. Überschreibt getRollenbezeichnung() so, dass sie "Azubi" zurückgibt. Das Enum Fachrichtung ist bereits vorgegeben, ihr müsst es nicht selbst bauen.

- ausbildungsjahr: int
- fachrichtung: Fachrichtung
+ Azubi(personalnummer, vorname, nachname, ausbildungsjahr, fachrichtung)
+ getRollenbezeichnung(): String
3

Ausbilder

Klasse extends PersonSehr leicht

Ausbilder erbt von Person und hat keine eigenen zusätzlichen Attribute. Der Konstruktor reicht alle drei Parameter einfach an super(...) weiter. Überschreibt getRollenbezeichnung() mit "Ausbilder".

+ Ausbilder(personalnummer, vorname, nachname)
+ getRollenbezeichnung(): String
Nicht jede Unterklasse braucht neue Attribute. Manchmal ist der einzige Unterschied, was getRollenbezeichnung() zurückgibt.
4

Bpu

Klasse extends PersonSehr leicht

Genau wie Ausbilder, nur dass getRollenbezeichnung() "BPU" zurückgibt. In eurem Bereich ist der BPU die zentrale Ansprechperson für jede Abteilung.

+ Bpu(personalnummer, vorname, nachname)
+ getRollenbezeichnung(): String

Teil 2 – Euer Modell: Praxiseinsätze

Ab hier baut ihr allein weiter, in eurem eigenen Package. Das Diagramm hat zwei Spalten: links die Klasse, die am Ende alles zusammenhält (Praxiseinsatz), rechts eine Kette aus drei Klassen, die aufeinander aufbauen (Abteilung → Praxisangebot → Praxisplatz). Baut die rechte Kette zuerst, dann Praxiseinsatz – dort braucht ihr Praxisangebot schon als Attributtyp.

5

Praxiseinsatzstatus

EnumLeicht

Legt das Enum Praxiseinsatzstatus mit genau vier Konstanten an.

GEPLANT
LAUFEND
ABGESCHLOSSEN
ABGESAGT
6

Protokollierbar

InterfaceLeicht

Legt das Interface Protokollierbar mit genau einer Methode an. Jede Klasse, die dieses Interface implementiert, verspricht eine Methode verlauf(), die eine Liste mit lesbaren Text-Zeilen liefert. Team Berichtsheft baut in seinem Bereich dasselbe Interface. Später soll die Oberfläche darüber für zwei Klassen, die inhaltlich nichts miteinander zu tun haben, eine einheitliche „Verlauf"-Ansicht zeigen können.

+ verlauf(): List<String>
7

Abteilung

KlasseLeicht

Eine ganz normale Klasse ohne Vererbung: Attribute, Konstruktor, Getter. Baut sie ohne Kürzel-Validierung – die Prüfung, ob kuerzel einem bestimmten Muster entspricht, kommt erst an Tag 2 dazu.

- id: int
- kuerzel: String
- beschreibung: String
- bpu: Bpu
+ Abteilung(id, kuerzel, beschreibung, bpu)
8

Praxisangebot

KlasseLeicht

Hält als Attribut eine Referenz auf Abteilung – das ist die Assoziation im Diagramm, kein extends. Baut sie ohne den Fachrichtungs-Filter, den es in der echten Anwendung gibt – der kommt an Tag 2 dazu.

- id: int
- abteilung: Abteilung
- beschreibung: String
- vorausgesetztesAusbildungsjahr: int
+ Praxisangebot(id, abteilung, beschreibung, vorausgesetztesAusbildungsjahr)
9

Praxisplatz

KlasseLeicht

Hält als Attribut eine Referenz auf Praxisangebot. Damit habt ihr die Kette Abteilung → Praxisangebot → Praxisplatz komplett.

- id: int
- angebot: Praxisangebot
- kapazitaet: int
- schwerpunkt: String
+ Praxisplatz(id, angebot, kapazitaet, schwerpunkt)
10

Praxiseinsatz

Klasse, implements ProtokollierbarMittel

Die zentrale Klasse eures Bereichs: Sie hält Referenzen auf Azubi, Praxisangebot und Ausbilder gleichzeitig und implementiert zusätzlich Protokollierbar. Der Konstruktor setzt alle vier Parameter direkt und zusätzlich status = Praxiseinsatzstatus.GEPLANT, obwohl status gar kein Parameter ist – jeder neue Praxiseinsatz startet als geplant. Für verlauf() reicht an Tag 1 eine Liste mit einem einzigen Eintrag, zum Beispiel List.of(azubi.getVollstaendigerName() + " – " + angebot.getBeschreibung() + " – " + status). Eine echte Historie kommt an Tag 2.

- nummer: int
- azubi: Azubi
- angebot: Praxisangebot
- zugewiesenDurch: Ausbilder
- status: Praxiseinsatzstatus
+ Praxiseinsatz(nummer, azubi, angebot, zugewiesenDurch)
+ verlauf(): List<String>
Das ist die einzige Klasse in eurem Modell mit drei Assoziationen gleichzeitig und einem Interface. Baut die Attribute zuerst, dann den Konstruktor, dann erst verlauf().

Wann seid ihr fertig?

  • Alle neun Typen aus diesem Dokument existieren mit exakt diesen Namen.
  • Das Projekt kompiliert ohne Fehler.
  • Praxiseinsatz implementiert Protokollierbar vollständig, @Override steht bei verlauf().
  • Ihr könnt an eurem eigenen Diagramm zeigen, welcher Pfeil Vererbung, welcher Interface-Umsetzung und welcher Assoziation ist.
  • Mindestens eine Person im Team kann erklären, warum euer Bereich – anders als Team Berichtsheft – keine eigene abstrakte Klasse hat.
Tag 2: Ihr bekommt eine geprüfte, ergänzte Fassung dieses Modells zurück – mit Praxisplanung, echten Statusübergängen (starten(), abschliessen(), absagen()) und einer echten Verlaufshistorie. Darauf arbeitet ihr weiter, unabhängig davon, wie weit ihr heute gekommen seid.
